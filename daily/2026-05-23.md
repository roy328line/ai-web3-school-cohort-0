  - ## Open Agentic Economy｜核心整理
 
  - ### 什麼是 Open Agentic Economy？
 
  - Open Agentic Economy 指的是一個 AI Agent 可以作為獨立經濟參與者的生態——Agent 能自主發現服務、協商任務、完成支付、留下可驗證記錄，而不只是「幫人操作介面的工具」。
 
  - 核心問題：當 Agent 不再只是助手，而是可以代表用戶（或自主）完成鏈上操作時，身份、授權、支付和責任邊界怎麼設計？
 
  - ### ERC-8004 / ERC-8183 方向
 
  - ERC-8004 / ERC-8183 是朝向「Agent 可讀取的鏈上能力描述」方向的標準探索：
 
  - - **Agent 發現服務**：Agent 如何找到它可以調用的工具和服務？需要機器可讀的能力聲明格式
    - - **可組合性**：不同 Agent 或協議之間如何互相調用、轉移任務、結算費用？
      - - **可驗證記錄**：Agent 做過什麼操作，如何在鏈上留下可審計的痕跡？
       
        - **與 MCP 的關係**：MCP（Model Context Protocol）解決的是「LLM 如何調用本地工具」，ERC-8004/8183 方向則更側重「鏈上環境下 Agent 如何發現和調用去中心化服務」。兩者互補，不互斥。
       
        - ### Builder Path 的核心判斷
       
        - 今日直播的 Builder Path 提供了一個判斷框架：
       
        - | 維度 | 問題 |
        - |------|------|
        - | 身份 | Agent 的身份是什麼？誰可以授權它？ |
        - | 權限 | Agent 能做什麼？邊界在哪裡？ |
        - | 支付 | Agent 的服務費如何結算？誰承擔？ |
        - | 記錄 | Agent 的行為如何被審計和追責？ |
       
        - ## Agent Wallet 深讀｜核心整理
       
        - ### 第一性原理
       
        - > 控制權不能交給一個概率系統。Agent 只能拿到可驗證、可限制、可撤銷的行動空間。
          >
          > 這句話的精髓：不是「不相信 AI」，而是承認 Agent 的判斷來自模型 + 上下文 + 工具返回值，任何一環都可能出錯。所以 Agent 的行動空間必須被系統規則包起來，而不是靠 Agent 自律。
          >
          > ### 核心架構：六層防禦
          >
          > Agent 安全執行鏈上操作的防禦層次：
          >
          > 1. **Policy**：定義 Agent 能做什麼（白名單合約、每日額度、允許的方法）
          > 2. 2. **Session Key**：臨時有限權限鑰匙，限時間、限額度、限目標、可撤銷
          >    3. 3. **Guard**：確定性攔截層，執行前用規則再次檢查（不符合就拒絕）
          >       4. 4. **Simulation**：交易發出前預演結果，把影響翻譯成人能看懂的語言
          >          5. 5. **Human Check**：高風險動作回到用戶確認（分層：低風險自動、中風險確認、高風險拒絕）
          >             6. 6. **Revocation**：隨時可撤銷的入口，系統也能在異常時自動收緊
          >               
          >                7. ### 關鍵概念對比
          >               
          >                8. | 概念 | 定義 | AI × Web3 意義 |
          >                9. |------|------|----------------|
          >                10. | EOA | 私鑰控制的普通帳戶 | Agent 不應直接持有 |
          >                11. | Smart Account | 合約定義的可編程帳戶 | Agent 的執行邊界容器 |

          >                13. | Policy | 系統可檢查的規則集 | 把「允許」變成可驗證條件 |
          >                14. | Guard | 執行前的確定性攔截 | 兜底防線，不依賴 Agent 自律 |
          >                15. | Simulation | 交易前結果預演 | 讓人看懂在批准什麼 |
          >               
          >                16. ### Account Abstraction（ERC-4337）補充
          >               
          >                17. ERC-4337 的核心流程：
          > ```
          > 用戶/Agent → 生成 UserOperation
          > → Bundler 收集並提交
          > → EntryPoint 合約驗證
          > → Smart Account 執行
          > → Paymaster（可選）承擔 gas
          > ```
          >
          > 對 Agent 場景的意義：Session Key 的限制條件就是寫在 Smart Account 的驗證邏輯裡，Bundler 和 EntryPoint 確保規則被執行，不由 Agent 自行決定。
          >
          > ## 個人發想
          >
          > 今天最大的收穫是理解了 **「Open Agentic Economy 的核心不是讓 Agent 更自由，而是讓 Agent 的自由被規則包起來」**。
          >
          > ERC-8004/8183 的意義在於：當 Agent 作為獨立經濟參與者出現時，整個生態需要一套共同語言——不只是技術標準，也是責任邊界和信任機制的標準化。
          >
          > Builder 的真正機會，不是做一個「聰明的 Agent」，而是做一個「可信任的 Agent 基礎設施」——讓用戶願意把部分授權交出去，因為規則足夠清楚、撤銷足夠方便、記錄足夠透明。
