## 模組 A｜AI 基礎：Agent 深度整理

### 第一性原理

> Agent 不是自主權本身，而是被約束的執行循環。目標、工具、狀態、權限和停止條件缺一不可。

一個只有模型和工具的 Agent，像是沒有停止條件的腳本執行器。真正可用的 Agent 必須知道：能做什麼、不能做什麼、做完如何驗證、失敗如何停止、誰能審計它做過的事。

---

### A-4. Agent 核心技術組件

#### Tool Use（工具調用）

讓 Agent 從「會回答」變成「能做事」。但工具也讓 Prompt Injection 和模型誤判變得危險。

工具設計必須明確的六個維度：

| 維度 | 說明 |
|------|------|
| 輸入 schema | 輸入格式和型別限制 |
| 權限範圍 | 哪些資源可以存取 |
| 是否只讀 | 讀取 vs 寫入的風險差距 |
| 外部副作用 | 是否改變鏈外狀態（如發送 Email、執行交易） |
| 記錄方式 | 調用前後如何記錄（可審計） |
| 人工確認觸發條件 | 哪些調用必須暫停讓用戶確認 |

**個人發想**：一個能「讀網頁」的 Agent 被惡意網頁影響，和一個能「寫入系統」的 Agent 被惡意網頁影響，後果完全不同。這就是為什麼 Tool Use 的權限分級比工具能力本身更重要。

#### Planning（規劃）

模型生成的計劃是**候選路線，不是授權**。

一個好的 Agent plan 應該能暴露：
- 每一步需要什麼工具
- 每一步是讀還是寫
- 哪些步驟可自動執行，哪些需要用戶確認
- 失敗時是否可重試
- 最終如何驗證任務完成

越靠近高風險動作（鏈上寫入、資產轉移），計劃越需要被系統規則拆開逐步檢查，而不是一次批准整條計劃鏈。

#### State（狀態管理）

State 是 Agent 當前任務的完整快照：用戶目標、已完成步驟、工具返回值、錯誤記錄、預算消耗、確認記錄、最終輸出。

常見問題：許多 Demo 只把 State 放在 prompt history 裡。生產系統需要**可查詢、可恢復、可審計**的外置 State，否則無法回答：
- Agent 為什麼調用了這個工具？
- 它是否已拿到用戶確認？
- 哪一步開始偏離目標？

在 AI x Web3 場景，State 還應記錄：環境（鏈 ID、區塊高度）、版本、工具調用結果、確認請求、撤銷事件。

#### Reflection（反思）

讓 Agent 檢查中間結果並修正下一步。

關鍵限制：**自我檢查可以提高質量，確定性檢查才能承載風險。**

Reflection 仍由模型完成，模型可能給自己的錯誤找理由。涉及鏈上寫入、授權、支付等動作，Reflection 只能作為輔助診斷，不能替代外部驗證 + 人工確認。

#### Multi-Agent

多個 Agent 分工協作（研究 / 開發 / 安全審查 / 執行）能處理複雜工作流，但會放大協調問題：
- 上下文傳遞丟失
- 責任邊界模糊
- 一個 Agent 的錯誤被另一個當成事實
- 工具權限擴散

**關鍵判斷問題**：多個 Agent 是否真的減少複雜度？如果只是把一個不清楚的流程拆成多個不清楚的角色，系統會更難調試，風險更難定位。

### AI x Web3 Agent 架構（八步模型）

從 Tool Use / State 視角深化昨日整理的架構：

```
用戶提出目標和約束
  ↓
Agent 讀取上下文 → 生成計劃（候選路線，非授權）
  ↓
系統拆分：只讀步驟 vs 候選寫入步驟
  ↓
只讀工具自動執行（RPC 查詢、ABI 讀取、合約 view 函數）
  ↓
寫入工具進 policy 檢查（工具 schema 校驗、權限確認）
  ↓
Simulation 展示鏈上影響（讓用戶看到結果再決定）
  ↓
用戶確認高風險動作（Human-in-the-loop）
  ↓
Wallet / Smart Account 執行
  ↓
日誌記錄每一步和最終狀態（可審計 State）
```

**最危險的設計**：模糊目標 + 廣泛工具 + 長期記憶 + 大額資產權限 同時存在。

### Hermes Agent 入門（今日預習）

今晚直播主題。根據課程資料，Hermes 的核心設計重點：

- **Skills（技能）**：可復用的高層指令集，支持自動發現和動態生成。讓 Agent 從「一次性任務」變成「可持續能力積累」。
- **Long-term Memory（長期記憶）**：跨 session 存儲與召回用戶偏好、項目配置、歷史決策。
- **Tool Calling**：模型輸出結構化請求 → 框架執行 → 回傳結果，是 Agent「從說話到做事」的核心機制。
- **Tracing**：可視化 Agent 執行鏈，讓每一步工具調用、計劃節點、用戶確認都可查。
- **Guardrails**：輸入輸出驗證規則，不符合規則則中止執行。防止模型輸出直接進入執行層。
- **Handoff**：子任務完成後移交控制權，是 Multi-Agent 系統的協調機制。
- **Error Recovery**：執行失敗時的重試、回退和人工介入，是系統穩健性的底線。
