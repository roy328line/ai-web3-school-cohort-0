## Machine Payment｜核心整理

### 什麼是 Machine Payment？

Machine Payment 討論的是 Agent、API、服務和錢包之間如何自動完成**報價、授權、付款、收據和預算控制**。

核心問題不是「AI 會花錢」，而是：
- 誰授權？
- 價格是多少？
- 什麼時候扣款？
- 失敗怎麼退？
- 收據如何驗證？
- 預算如何不被濫用？

> Machine Payment 的核心，是把「付款意圖」和「實際結算」拆開，並讓每一步都有憑證。

---

### 第一性原理

> Agent 不應該擁有無限支付能力，只應該拿到具體任務、預算和收款方範圍內的支付權限。

支付是執行能力的一部分。一旦 Agent 可以付款，它就可以消耗用戶資金、觸發鏈上狀態變化，或被惡意上下文誘導。

三個核心原則：
1. **預算先於執行**：沒有預算邊界，就沒有安全自動支付
2. **報價必須可比較**：Agent 要知道價格、幣種、有效期、服務範圍和退款條件
3. **收據必須可驗證**：付款後要能證明付給誰、為什麼付、交付了什麼

### 知識節點整理

#### Stablecoin Payment（穩定幣支付）

- 穩定幣適合機器支付：價格穩定、結算快、可編程、容易被合約驗證
- Agent 支付 API 費時通常不希望幣價劇烈波動
- 需區分「計價幣種」和「結算幣種」（服務方可用 USD 報價，用戶用 USDC 支付）
- 判斷穩定幣方案的四件事：流動性、收款方接受度、失敗是否消耗 gas、是否需提前 approve

#### Budget（預算）

- Budget 是用戶授權給 Agent 的最大支出範圍
- 可按時間、任務、服務方、幣種和額度定義（如「今天最多花 5 USDC」）
- 預算要進入 wallet policy、session key 或後端帳本，不能只存在聊天裡
- 建議多層拆分：全局預算 → 任務預算 → 單次上限 → 服務方上限 → 緊急停止

常見錯誤：只設總預算，沒有設頻率和服務方範圍 → Agent 可能把全部預算花在一個異常報價上

#### Quote（報價）

- 一個合格 quote 至少包含：服務內容、價格、幣種、收款地址、有效期、交付條件、退款條件、quote id
- Quote 的有效期很重要：過期 quote 不能繼續使用
- Quote 應該能被簽名或驗證來源

#### Payment Intent（付款意圖）

- Payment Intent 表達「用戶或 Agent 想為某個具體服務付款」，但不等同於已結算
- 應綁定：任務、金額、收款方、有效期和可接受結果
- 讓後續 payment、escrow 和 receipt 有上下文依據

#### x402

- 把 HTTP 402 Payment Required 變成互聯網原生支付流程
- 流程：客戶端請求 → 服務返回付款要求 → 客戶端完成支付 → 帶付款證明重新請求
- 適合：API、數據、內容和 agent service 的小額按次付費
- 注意：x402 只解決付款協議的一部分，高價值服務仍需 escrow、receipt 和 dispute 機制

#### MPP（Machine Payments Protocol）

- 關注機器與機器之間的支付協商和結算
- 協議化流程：服務發現 → 獲取報價 → 授權支付 → 結算 → 返回收據
- 很多步驟可鏈下完成，鏈上只承擔最終結算、抵押、收據錨定或爭議證據

#### Subscription（訂閱）

- 持續服務的支付模型（如每月 API 包、持續監控）
- 訂閱比單次付款更需要撤銷能力
- 要小心「靜默續費」：系統必須讓用戶知道何時續費、最高預算、是否能漲價
- 建議與智能帳戶 policy 結合，不要通過無限 allowance 實現訂閱

#### Micropayment（微支付）

- 適合高頻、小額、自動化服務（如每次搜索、輕量推理、網頁抓取）
- 每次都上鏈結算成本可能超過服務本身
- 實際系統需要：L2、payment channel、批量結算、預付餘額或鏈下帳本 + 鏈上最終結算
- 設計前先算經濟帳：單次服務價值 vs 鏈上手續費 vs 失敗率 vs 對帳成本

### Machine Payment 在 AI x Web3 的位置

Machine Payment 是 **Agentic Commerce 的基礎**。

可用鏈路：
```
用戶授權預算 → Agent 獲取 quote → 系統檢查 policy → 付款進入 escrow 或直接結算 → 服務交付 → 收據留下證據
```

支付能力必須和 Agent Wallet、Policy、Settlement、Receipt 連接起來。

---

### 關鍵概念對比

| 概念 | 定義 | 重點 |
|------|------|------|
| Budget | 用戶授權的最大支出範圍 | 必須進入 policy，不能只在聊天中 |
| Quote | 服務方給 Agent 的可執行報價 | 需有有效期、簽名、quote id |
| Payment Intent | 付款意圖，未等同結算 | 綁定任務、金額、收款方 |
| x402 | HTTP 402 付款協議 | 適合小額按次付費 |
| Micropayment | 高頻小額支付 | 需考慮批量結算降低成本 |
| Subscription | 持續服務訂閱 | 需要明確撤銷機制 |

### 個人發想

今天最大的收穫是理解了**機器支付不是技術問題，而是信任問題**。

傳統支付（人對人）依賴法律合約和信任關係。但 Agent 對服務的支付，發生在毫秒之間，沒有時間讓人審查——所以這個信任必須被系統化、協議化、可驗證化。

x402 的概念讓我特別有感：把付款邏輯放回 HTTP 語義本身，Agent 處理 402 就像處理 401 一樣自然——這才是真正的「互聯網原生 AI 支付」。

Builder 的機會在於：設計一個「Agent 可以信任、用戶可以控制、服務方可以驗證」的支付基礎設施，讓 Agentic Commerce 從 Demo 走向真實商業場景。
