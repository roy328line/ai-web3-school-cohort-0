## 模組 A｜AI 基礎：核心知識整理

### A-1. LLM 大語言模型

**第一性原理**：LLM 生成的是「概率上合理的輸出」，不是天然可信的事實。

**核心知識點：**

- LLM 是概率生成模型：給定文本 → 預測最合理的下一個 token 序列
- **Token**：模型處理的基本單位，不等於一個字或詞。影響上下文容量、成本和信息完整性
- **Embedding**：把文字/代碼映射成向量，衡量語義相近程度，適合搜索/RAG，但不能判斷「結論是否正確」
- **Hallucination（幻覺）**：模型自信地生成不真實內容，在執行型系統裡幻覺會從「答錯」變成「做錯」

**LLM 擅長 vs 不擅長：**

| 擅長 | 不擅長 |
|------|--------|
| 語言理解、代碼生成、推理 | 精確事實記憶 |
| 文檔摘要、計劃生成 | 確定性計算 |
| 自然語言轉結構化格式 | 跨會話狀態保持 |

**在 AI x Web3 的位置**：LLM 是「理解和生成層」，越靠近執行層，越需要外部數據 + 確定性規則 + 人工校驗。

**個人發想**：幻覺在 Web3 場景特別危險。若 Agent 把幻覺出來的合約地址或參數直接送進交易，會造成資產損失。Simulation 和 Human-in-the-loop 不只是安全功能，而是系統基礎。

---

### A-2. Prompt 提示詞

**第一性原理**：Prompt 是軟約束，不是安全邊界。真正的邊界必須由代碼、權限、校驗和審計承擔。

**核心知識點：**

- **Instruction**：任務規則，回答「角色/目標/禁止事項/不確定時處理方式/輸出格式」
- **Few-shot**：放示例讓模型模仿，適合難以一句話描述的場景。示例需要跟 eval 一起維護
- **Structured Output**：讓模型按固定格式輸出（如 JSON），讓後續代碼可校驗、記錄、回歸測試
- **Prompt Injection**：攻擊者通過外部文檔讓模型忽略規則或調用危險工具，Agent 場景尤其危險

**好 Prompt 的四段結構**：任務目標 → 可用輸入 → 禁止行為 → 輸出格式和失敗格式

**個人發想**：Prompt 只是「說清楚任務」，系統才是「執行約束」。真正安全需要代碼層 allowlist + 工具調用前參數校驗 + 高風險動作強制走 human approval。

---

### A-3. Context 上下文

**第一性原理**：模型只能基於它看見的上下文行動；系統必須決定什麼能進上下文、帶著什麼身份進去、過期後怎麼退出。

**核心知識點：**

- **Context Window**：模型一次請求的最大信息範圍，長上下文容易「看見但沒抓住重點」
- **Context Engineering**：設計上下文進入模型的方式——選擇數據源、排序、裁剪、標注來源、隔離不可信內容
- **Memory**：跨請求保留信息。記憶不能替代實時授權，過去允許某動作不代表現在仍然允許
- **Knowledge Base**：外部可檢索知識庫，需維護來源、更新時間、版本和廢棄狀態

**穩健 Agent 上下文五層結構**：
1. 指令層（系統規則、禁止事項）
2. 任務層（用戶目標、本次會話參數）
3. 事實層（鏈上狀態、工具結果、simulation）
4. 知識層（文檔、標準、歷史案例）
5. 記憶層（用戶偏好、項目配置）

**個人發想**：不可信外部內容必須跟系統指令層隔離，否則外部「假指令」讓模型誤以為是授權操作。這正是 Prompt Injection 的根本機制。

---

### A-4. Agent 智能體

**第一性原理**：Agent 不是自主權本身，而是被約束的執行循環。目標、工具、狀態、權限和停止條件缺一不可。

**核心知識點：**

- **Tool Use（工具調用）**：讓 Agent 從「會回答」變成「能做事」。需明確輸入 schema、權限範圍、是否只讀、是否有外部副作用
- **Planning（規劃）**：把目標拆成步驟，計劃是候選路線不是授權，越靠近高風險動作越需要規則拆開檢查
- **State（狀態）**：任務當前狀態，必須外置可查、可恢復、可審計，不能只藏在模型上下文裡
- **Reflection（反思）**：讓 Agent 檢查中間結果並修正，能提高品質但不能替代外部驗證
- **Multi-Agent**：多 Agent 分工協作，要警惕上下文丟失、責任模糊、工具權限擴散

**穩定 AI x Web3 Agent 八步架構**：
用戶提出目標 → Agent 讀上下文生成計劃 → 只讀工具自動執行 → 寫入工具進 policy 檢查 → Simulation 展示鏈上影響 → 用戶確認 → Wallet/Smart Account 執行 → 日誌記錄

**個人發想**：Agent 最危險的設計是「模糊目標 + 廣泛工具 + 大額資產權限」三者並存。未來設計 AI x Web3 產品時，最小化 Agent 初始權限應該是設計原則，不是後期補丁。

## 模組 B｜Web3 基礎：核心知識整理

### B-1. 錢包（Wallet）

**第一性原理**：錢包管理的不是「帳號資料」，而是鏈上控制權。

**核心知識點：**

- **三類動作風險差異**：
  1. 連接錢包：只讀取地址和網絡，不等於可以動用資產
  2. 簽名消息：證明控制某個地址，可能用於登錄/授權
  3. 發送交易：請求改變鏈上狀態，消耗 gas

- **EOA**：由私鑰控制，簡單通用，但私鑰丟失難恢復、權限難以細分
- **Mnemonic（助記詞）**：任何應用/Agent/表單要求輸入助記詞都應默認視為危險
- **Transaction（交易）**：簽名 → 廣播 → 打包 → 執行的完整流程，每步都可能失敗
- **Gas**：鏈上執行成本，費用資產、失敗是否仍消耗、如何重試——前端必須展示
- **Explorer（區塊瀏覽器）**：查看鏈上事實的窗口，AI 產品引用鏈上信息時最好給 explorer 連結讓用戶自行驗證

**個人發想**：設計產品時，UI 必須把連接/簽名/交易三種動作的文案、按鈕狀態、確認提示設計得截然不同，讓用戶能即時感知風險等級。

---

### B-2. 智能合約（Smart Contract）

**第一性原理**：合約的價值來自可驗證執行，不來自「代碼看起來很聰明」。

**核心知識點：**

- **Solidity**：EVM 生態最常見合約語言。關鍵：storage（鏈上狀態）、msg.sender、modifier、event、revert
- **EVM**：合約執行環境，決定代碼如何被部署、調用、計費和隔離
- **ABI**：合約接口說明，是機器可讀接口不是安全說明書——告訴你能調用什麼，不保證調用是否安全
- **Event**：合約向外部留下的可索引日誌，是前端、索引器、分析工具的數據來源
- **Upgrade（升級）**：引入管理員權限和信任問題。評估四問：誰有升級權限？用戶能看到提案嗎？能改變資產邏輯嗎？密鑰洩露的最壞結果？

**完整鏈上調用流程**：
前端讀取錢包地址 → ABI 編碼調用數據 → 錢包讓用戶確認 → 廣播到 RPC → 驗證者打包進區塊 → EVM 執行 → 更新狀態/發出 event → 前端監聽回執 → 索引器更新數據層

**個人發想**：ABI 只告訴你能調用什麼，不保證安全。Agent 不理解地址是否可信、金額單位是否正確、function 是否被暫停。Simulation + 人工確認是必要的。

---

### B-3. 帳戶抽象（Account Abstraction）

**第一性原理**：當帳戶本身可以編程，權限就可以從「有私鑰/沒私鑰」變成「在什麼條件下允許什麼動作」。

**核心知識點：**

- **ERC-4337**：以太坊帳戶抽象標準。UserOperation → Bundler 收集打包 → EntryPoint 調用智能帳戶驗證和執行 → Paymaster 可選贊助 gas
- **Smart Account**：由合約控制，支持多簽、社交恢復、批量交易、細粒度權限。合約 bug 和升級邏輯會成為帳戶風險
- **Bundler**：收集 UserOperation 並提交到 EntryPoint，Bundler 不穩定則用戶操作卡住
- **Paymaster**：讓第三方贊助 gas，需設計風控防止 spam
- **Session Key**：給 Agent 的臨時權限——限定時間、合約、方法、額度、鏈

**Session Key 的價值**：不希望 Agent 每次都打斷用戶簽名，也不希望 Agent 擁有無限權限。Session Key 提供中間狀態：Agent 可自動執行低風險動作，高風險動作仍需用戶確認。

**個人發想**：Session Key 的設計哲學是最小權限原則（Principle of Least Privilege）。比喻：不是「給 AI 一把主鑰匙」，而是「給 AI 一張限時限額的員工門禁卡」。

## 模組 A + B 交叉發想：AI x Web3 的核心設計張力

**AI 側最大的不確定性**：模型輸出不可靠（幻覺、注入、推理漂移）
**Web3 側最大的不可逆性**：鏈上執行不能撤回（交易上鏈、合約調用、資產轉移）

這兩個特性放在一起，創造了系統設計上的核心張力：**需要用不可靠的推理引擎來驅動不可逆的執行系統。**

解法不是讓 AI 更可靠或讓 Web3 更可逆，而是在中間建立清晰的驗證層和授權邊界：

| 機制 | 解決的問題 |
|------|-----------|
| Simulation（模擬執行） | 在真正上鏈前先看到結果 |
| Structured Output（結構化輸出） | 讓 AI 輸出機器可校驗的格式 |
| Session Key（限時限額授權） | 給 Agent 剛好夠用的權限，不多 |
| Human-in-the-loop（人工確認） | 高風險動作必須暫停讓人決定 |
| Audit Log（審計日誌） | 每一步都留下可回溯的記錄 |
