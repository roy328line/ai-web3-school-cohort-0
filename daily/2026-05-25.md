## Settlement & Escrow｜核心整理

### 什麼是 Settlement & Escrow？

Settlement & Escrow 解決的是 Agent 經濟裡的「錢什麼時候釋放、服務怎麼算完成、失敗怎麼退款、爭議怎麼處理」。它把支付從一次轉帳變成完整交易流程。

核心問題不是「如何付款」，而是：

- 付款後，服務如何確認完成？
- 交付物如何驗證真實性？
- 失敗時誰來觸發退款？
- 雙方有爭議時誰有裁決權？
- 整個流程是否可審計、可驗證？

Settlement & Escrow 的核心，是把任務、交付、驗收和付款綁定成可驗證的狀態機流程。

---

### 第一性原理

> **自動化交易必須有明確完成條件，否則支付就無法安全自動化。**

如果任務定義模糊，Agent 和服務方就很難判斷何時完成、誰來確認、失敗怎麼辦。Escrow 設計首先要定義狀態機，而不是先寫付款代碼。

三個設計要求：

- **資金狀態要清楚**：pending、locked、released、refunded、disputed
- **交付證明要可保存**：結果、hash、日誌、模型輸出、交易哈希都可能成為證據
- **爭議流程要提前設計**：不要等失敗後才討論誰有權判定

### 知識節點整理

#### Escrow（託管）
- Escrow 是把資金暫時鎖在合約或可信帳戶裡，等待交付條件滿足後再釋放
- 在 Agent 場景裡，escrow 可以綁定任務 ID、付款方、服務方、金額、截止時間、驗收規則和退款條件
- 最小狀態機：Created → Funded → Delivered → Accepted → Released（或 Refunded / Disputed）
- 關鍵：好的 escrow 先定義業務流程，再定義資金流

#### Receipt（收據）
- Receipt 不只是「已付款」，應記錄：付款方、收款方、金額、幣種、任務 ID、quote id、時間、交易 hash、服務結果引用、驗收狀態
- Receipt 還會成為 reputation 的輸入：是否按時付款、是否按時交付、任務是否被退款
- 最好同時服務人和機器：人能看懂，機器能解析

#### Delivery Proof（交付證明）
- 證明服務方確實交付了某個結果
- 可以是：文件 hash、API 返回日誌、模型輸出簽名、鏈上 event、TEE attestation、人工驗收記錄
- 要避免「結果存在但不可驗證」：必須能和原始任務對應
- 不同任務需要不同 proof：API 調用用請求/響應 hash，代碼任務用 commit hash + 測試結果

#### Acceptance（驗收）
- 付款方或規則系統確認「交付滿足要求」的動作
- 可以由用戶點擊確認，也可以由自動 evaluator 判斷
- 應拆成可檢查條件：字段是否完整、測試是否通過、hash 是否匹配
- 高價值任務建議：AI 初審 + challenge window + 人工復核的組合

#### Refund（退款）
- 退款規則要在任務開始前寫清楚：誰能觸發、多久後可以觸發、是否扣除部分費用
- 常見退款觸發：服務方超時、交付格式錯誤、驗收失敗、任務被取消、quote 過期
- 要考慮部分交付的情況：服務方完成了一部分，是否按比例付款？

#### Dispute（爭議）
- 爭議可由：人工仲裁、多簽、DAO、optimistic challenge、第三方 evaluator、鏈上規則處理
- 設計至少要回答：誰能發起、發起成本是多少、證據提交格式、誰有裁決權、是否可申訴
- 爭議記錄也是聲譽系統的重要輸入：頻繁進入 dispute 的服務方或客戶，後續交易應被看到

#### Evaluator（評估者）
- 判斷交付是否合格的角色或系統
- 可以是：腳本、測試套件、模型、人工審核員、多個驗證者、鏈上合約
- AI evaluator 適合做初步判斷，但高價值任務需要可復核證據和挑戰機制
- 組合 evaluator：腳本檢查格式 → 測試套件檢查功能 → AI 檢查語義 → 人類處理爭議

#### ERC-8183
- 圍繞 agentic commerce 的草案標準，關注 Agent 任務、支付、escrow、交付和驗證如何形成統一流程
- 把 agent commerce 從「隨便發一筆錢」推進到更結構化的交易模型
- 重點不是馬上採用標準，而是學習它試圖解決的抽象：Agent 經濟的交易是狀態轉換，不是單純 token transfer
- 與 ERC-8004 互補：ERC-8004 偏 Agent 身份/聲譽；ERC-8183 偏任務/支付/託管/交付

### Settlement & Escrow 在 AI x Web3 的位置

Settlement & Escrow 是 Machine Payment 的後半段：

- Machine Payment 解決「如何付款」
- Settlement & Escrow 解決「付款後如何確認價值交換完成」

完整鏈路：**用戶鎖定預算 → 服務方交付 + 提交 Delivery Proof → Evaluator 驗收 → 釋放付款 → 生成 Receipt**

失敗路徑：**超時 / 驗收失敗 → Refund → 記入 Receipt → 影響聲譽**

---

### 關鍵概念對比

| 概念 | 定義 | 重點 |
|------|------|------|
| Escrow | 鎖定資金直到條件滿足 | 先定義業務流程，再定義資金流 |
| Receipt | 支付與交付的完整憑證 | 同時服務人和機器可讀 |
| Delivery Proof | 服務方交付的可驗證證明 | 必須能和原始任務對應 |
| Acceptance | 確認交付滿足要求的動作 | 可自動也可人工，需提前定義標準 |
| Refund | 失敗或取消時退回資金 | 規則必須在開始前寫清楚 |
| Dispute | 雙方對交付的分歧處理 | 需要有成本、有裁決機制、可記錄 |
| Evaluator | 判斷交付是否合格的系統 | 建議組合多層評估 |
| ERC-8183 | Agentic Commerce 草案標準 | 任務生命週期的狀態轉換模型 |

### 最小實踐

設計一個「Agent 付費生成報告」的 escrow 流程：

1. 用戶鎖定 2 USDC
2. 服務方承諾 10 分鐘內交付報告
3. 報告提交後記錄 IPFS hash 或文件 hash
4. Evaluator 檢查報告是否包含指定字段
5. 通過則釋放付款，失敗或超時則退款
6. 有爭議時進入人工或多簽仲裁

### 個人發想

今天最大的收穫是理解了 Escrow 的本質不是「鎖錢」，而是「把整個業務流程狀態機化」。

一個沒有明確完成條件的合約，就算有 escrow 機制也沒有意義——因為無法判斷何時釋放、何時退款。這讓我重新理解了為什麼鏈上合約設計首先是業務流程設計，而不是技術實現。

ERC-8183 的思路也很有啟發：它把 Agent 交易從「轉帳」提升到「狀態轉換」的維度，這樣整個流程就可以被系統理解和驗證，而不是依賴雙方的善意。

Evaluator 的設計是我覺得最有挑戰性的部分：一個 AI 做初審是否公平？如果 evaluator 本身有偏見怎麼辦？「AI 初審 + challenge window + 人工復核」的組合似乎是目前最合理的平衡——讓自動化處理低風險任務，讓人類處理邊緣案例。

Builder 的機會在於：設計一個「狀態透明、proof 可驗證、dispute 有成本但可處理」的 Agentic Escrow 系統，讓 AI Agent 之間的委托和交易真正可信任。
