    - ## Week 1 整體複習｜核心脈絡整理
   
    - 本週學習的核心主軸是：**用不可靠的推理引擎（AI）驅動不可逆的執行系統（Web3）**——這是整個課程的核心設計張力。
   
    - ### AI 基礎脈絡（Day 1–5 整合）
   
    - 五天下來，AI 這條線從「模型是什麼」走到「Agent 怎麼安全執行」：
   
    - - **LLM**：概率生成，不是事實查詢引擎。Hallucination 在 Web3 裡不是答錯，是做錯。
      - - **Prompt**：軟約束，不是安全邊界。好 Prompt 四段式（目標 → 輸入 → 禁止 → 輸出格式）是起點，系統安全需要代碼層 allowlist + 人工確認。
        - - **Context**：五層結構（指令 → 任務 → 事實 → 知識 → 記憶），不可信外部內容必須與指令層隔離。
          - - **Agent**：受約束的執行循環，不是自主體。最危險設計：「模糊目標 + 廣泛工具 + 大額資產權限」三者並存。
            - - **Frameworks**：先理解工作流，再決定用不用框架。LangChain（組件庫）、LangGraph（狀態機）、OpenAI Agents SDK（工程框架）、Hermes（穩定 tool calling 生態）各有邊界。
             
              - ### Web3 基礎脈絡
             
              - - **Wallet**：連接、簽名、交易三類動作風險截然不同。助記詞 = 最高危資訊。
                - - **Smart Contract**：ABI 告訴你能調用什麼，不保證調用是否安全。一次完整鏈上調用是 9 步流程。
                  - - **Account Abstraction**：ERC-4337 讓帳戶可編程。Session Key = AI Agent 上鏈的安全關鍵基礎（最小權限原則，限時限額）。
                   
                    - ### AI × Web3 交叉設計原則
                   
                    - 1. **AI 解釋 ≠ AI 授權；AI 建議 ≠ AI 執行**
                      2. 2. **每個執行動作需要可追溯的確認記錄**
                         3. 3. **Simulation + Structured Output + Session Key + Human-in-the-loop + Audit Log** 是解法組合
                            4. 4. 分工：AI Framework 管理 prompt/tools/state/eval/trace；Web3 基礎設施管理帳戶/簽名/合約/鏈上狀態；產品層定義確認流程和失敗處理

                                                    - ## 個人學習感受（例會分享準備）
                                                   
                                                    - Week 1 最大的收穫不是學了哪些技術點，而是建立了一個**判斷框架**：
                                                   
                                                    - > 「任何 AI 輸出進入 Web3 執行層之前，都需要一道人工確認節點。AI 降低了操作門檻，但不能降低安全標準。」
                                                      >
                                                      > 這個判斷框架讓我能夠評估任何 AI × Web3 應用的設計是否合理——不是看它功能多豐富，而是看它在哪裡暫停、讓人確認、留下記錄。
                                                      >
                                                      > Hermes 的深入學習讓我意識到：**框架選擇和模型選擇是兩個不同維度**，一個工具調用夠穩定的模型可以減少框架層的 guard，這對鏈上場景的系統設計成本影響很大。
